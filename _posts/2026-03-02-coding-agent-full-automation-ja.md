## Vibe Codingから「自律作業」へ：Coding Agentで開発・調査・執筆を自動化するまでの全記録

> この記事では感想を語らない。「映画を観ながらAIが勝手に開発を回している」状態にどうやって辿り着いたか、その過程を全部書く。記事の最後にSkillsのソースファイルを全公開するので、そのままコピーして使える。

---

### 冒頭：ある土曜の夜の風景

2月最後の土曜日、夜9時過ぎ。

テレビでは映画が流れている。手元のiPadではストラテジーゲーム。そしてすぐ横のノートPCのターミナルでは、Claude CodeとCodexが「議論」を繰り広げている。

Claude Codeがあるプロダクトのコードフレームワークを一版書き上げると、Codexがレビューして「アーキテクチャに問題あり、ここを直せ」と返す。Claude Codeが自分で修正し、再提出。Codexが再レビュー。

私はたまにターミナルをチラ見して、正常に回っているか確認するだけ。

別のターミナルウィンドウでは、その日の昼間に `/log` で投げ込んだ断片的な作業メモ——「病院で腰と背中を診てもらった」「あるブランドコンサルタントの方法論を調べた」「パートナー企業との戦略提携案を整理した」——が、すでに構造化された業務日誌に自動展開されていた。日誌の末尾には「SNS投稿に使える視点」3本と「記事に深掘りできるテーマ」2本が自動で抽出されている。

これは効率が良いという話ではない。**働き方そのものが変わった**という話だ。

1ヶ月前、私はまだAIとチャットウィンドウで一つずつやり取りし、手動でコンテキストをコピペし、一度に一つの仕事しか進められなかった。今はすべてが自動で回っている——開発にはAIが書いてAIがレビューする閉ループがあり、日誌には自動展開とコンテンツ抽出があり、記事は日誌の蓄積から自動生成するパイプラインがある。

この記事では、この1ヶ月間の変化を余すところなく書く。「AIとのチャット」から「AIが勝手に仕事をする」状態にどうやって辿り着いたか。天才的なアイデアなどない。日々の業務のなかで試行錯誤を繰り返した、ごく普通のプロセスだ。

記事の最後に、すべてのSkillsソースファイルを公開する。`/log`、`/tweet`、`/article`、`/plan`、`/develop`。コピーすればすぐ使える。

### 第一幕：CLIの対話画面で起きた「気づき」

Claude Code CLIに触れたきっかけは、いわゆるVibe Codingだった。

ブログを作りたい？AIと数回やり取りすれば形になる。マルチプラットフォームへの自動投稿ツールが欲しい？要件を伝えれば書いてくれる。以前なら「こういうの作りたいな」と思っても数ヶ月は放置していた。フレームワーク選定、環境構築、実装——どのステップにも摩擦がある。それが、AIと話しているうちにプロトタイプが動き始めた。

しかしVibe Codingは入口にすぎなかった。

本当に「これは違う」と感じたのは、plan modeの能力がコーディングの範囲を遥かに超えていると気づいたときだ。

ある日、海外展開しているブランドの事業戦略を進めていた。従来のやり方なら、静かな午後を確保してドキュメントを広げ、一から整理する。だがマネージャーとして、1日が会議で7つ8つに分断されるのが日常で、そんなまとまった時間は存在しない。

その日、新しい方法を試した。ブランドの背景情報、これまでの調査メモ、競合資料を、すべてmarkdownドキュメントとしてローカルの1つのディレクトリに入れた。そしてClaude Codeに「あなたは私の戦略コンサルタントです。これらの資料をもとに、新しい運営戦略を整理してください」と伝えた。

AIはまず全ファイルを自分で読み、いくつか質問してきた。「ターゲット顧客層は？」「予算の制約は？」「競合との核心的な差別化ポイントは？」

10分ほどのやり取りの後、アウトプットが始まった。

出てきたのは「ブランドポジショニングを明確にしましょう」「ユーザー体験を重視しましょう」といった一般論ではない。非常に具体的な戦略フレームワークだった——どの層を狙うか、どう差別化するか、価格設定はどうするか、KOL連携の方法、オンラインとオフラインの連動施策。

これには驚いた。

十分なcontext——一行の説明ではなく、ディレクトリ丸ごとの背景資料——を渡したことで、アウトプットの質が「チャット型AI」とは段違いになっていた。

**ここが転換点だった。AI Agentのアウトプット品質は、こちらが渡すcontextの品質で決まる。** AIの能力が足りないのではない。こちらが渡す情報が足りないのだ。

その日から、ローカルPC上にcontextの体系を意識的に構築し始めた。プロジェクトごとにディレクトリを作り、CLAUDE.mdにプロジェクトの背景と前提を書き、関連する調査ドキュメント、過去の意思決定記録、議事録を.mdファイルで蓄積する。

一見「余計なひと手間」に見えるこの習慣が、その後の膨大な時間を節約することになる。

### 第二幕：1ウィンドウから6ウィンドウへ

contextが蓄積されるにつれて、ある変化が明確になってきた。**スピードが桁違いになった。**

以前は新しいタスクを始めるたびに15〜20分かけて背景を説明していた。「うちの会社はこういう事業で、このプロジェクトの経緯はこうで、これまでにこういう施策を試した」と。今はディレクトリを開けば、AIが自分でCLAUDE.mdとプロジェクトファイルを読み、30秒で前回の続きに入れる。

こうなると、自然と同時に5つ、6つのClaude Codeウィンドウを開くようになった。

1つは海外ブランドの運営戦略。1つはパートナー企業との提携方針の整理。1つは自社AIプロダクトのイテレーション。1つは社外アドバイザーへの報告資料の準備。1つは業務日誌の作成……。

何に似ているかというと、一人で5〜6人のインターンを同時に見ているのに近い。それぞれが別のプロジェクトを担当し、自分はその間を行き来しながら進捗を確認し、フィードバックを出し、方向を調整する。かつてこの能力は「マネジメントスキル」と呼ばれていた。それがAI協働のなかで新しい活躍の場を見つけた。

実際の効率を1つの例で伝える。

あるパートナー企業の幹部が、繰り返し深い連携への意欲を示していた。この情報は私だけが知っていたわけではない。担当チームも把握していた。だが私が一歩踏み出さなければ、この組織の中でおそらく誰もこの件に手をつけなかった。

Claude Codeを使い、1時間もかからず、両社の事業情報、互いの強み、連携の方向性、具体的な推進ステップを整理した。戦略提携方針の初稿、一式完成。

そして思った——チームの他のメンバーは、同じ情報を手にしていながら、誰一人としてたった10分すら「AIに整理させてみよう」という一歩を踏み出していない。

能力の問題ではない。誰にでもできることだ。原因はもっと複雑で——「自分のKPIじゃない」「自分の担当範囲じゃない」「そもそもAIを業務に使う習慣がない」。

この出来事から、より深い気づきを得た。**AIは「何かをやる」コストをほぼゼロにした。しかし「これをやるべきだ」と判断し、自ら動き出す意志——これはAIにはどうにもできない。**

日本の組織には、この課題がより色濃く存在すると感じる。役割分担が明確で、「越境」することへの心理的ハードルが高い。報連相の文化は情報共有には優れているが、「誰かがやってくれるだろう」という受け身も生みやすい。

これからの時代に人に求められるのは、「AIを使えるかどうか」ではない。好奇心があるか。自分から手を動かす気概があるか。自分の職務範囲を越えて、「自分の仕事じゃないかもしれないが、やれば価値がある」ことに踏み出せるか。

これを「起動力」と呼んでいる。実行力はAIがすでに民主化した。次に差がつくのは、起動力だ。

もちろん、スピードが上がれば新しいボトルネックも現れる。

**AIのアウトプットが増えるほど、自分のレビューと判断がボトルネックになった。** 5〜6つのウィンドウが並行して方針書、ドキュメント、コードを吐き出す。すべてが私の検収を待っている。AIは「ゼロから70%」の効率を劇的に上げたが、「70%から100%」——つまり読む、判断する、調整する——この部分がむしろ最大の生産性ボトルネックになった。

片方にはAIが生成したN件のドキュメント。もう片方にはN人のメンバーが待つ1on1。どちらも「自分本人」がその場にいなければ進まない。

この問題にしばらく悩まされた。解決策は後述する。

### 第三幕：断片的な記録をコンテンツ資産に変える

毎日マルチスレッドで仕事をしていると、価値ある思考や経験が頭の中に散乱したまま記録する暇がなくなる。

これは切実な問題だ。朝の会議後に良いアイデアが浮かび、午後のプロジェクト推進で新しい知見を得て、夜にAIと試行錯誤して面白いワークフローを発見する。だが振り返りを書こうとする頃には、半分以上忘れている。

そこで、自分専用のコンテンツ生成Skillsを構築し始めた。

**Skillsとは何か？** 簡単に言えば、あらかじめ書いておいた一連の指示を `.claude/commands/` ディレクトリに保存しておくもの。Claude Codeで `/コマンド名` と入力するだけで即座に起動し、毎回ゼロから要件を伝え直す必要がない。

最初に作ったのは **`/log`** だ。

一日の仕事が終わったら、断片的なメモを投げ込む。「今日は何々をやった、こういう問題があった、こんなことを考えた」——数行でいい。雑でも構わない。`/log` が自動的に以下を行う。

1. `date` コマンドで正確な日付を取得（自分で曜日を計算する必要なし）
2. 断片的な内容を構造化された日誌に展開——成果、学び、課題、翌日の予定、考察
3. 「SNS投稿に使える視点」と「記事に深掘りできるテーマ」を自動抽出
4. `logs/YYYY-MM/YYYY-MM-DD.md` のパスに自動保存

最後のステップが重要だ。**日誌の末尾に、次のコンテンツの方向性が自動で提示される。** つまり日誌は単なる「記録」ではなく、将来の記事やSNS投稿の素材ライブラリを積み上げるプロセスになっている。

次に作ったのは **`/tweet`**。

SNS投稿は適当に数行書けばいいわけではない。プラットフォームごとに文字数制限も表現の慣習も全く異なる。`/tweet` のやり方はこうだ。

- テーマや視点を1つ指定する
- 自動的に `logs/` ディレクトリから関連素材（実体験やケース）を検索
- 同じテーマで2つの独立バージョンを生成：300字版（Note、LinkedIn向け）と140字版（X/Twitter向け）
- 短縮版は長文版の削りではなく、独立した書き起こし。短いコンテンツと長いコンテンツでは表現のロジックが根本的に異なるからだ

最後に **`/article`**。

日誌が1〜2週間分蓄積されると、そこに自然と深掘りすべきテーマが数本沈殿している。`/article` がやることは以下だ。

- `logs/` と `summaries/` から関連素材を自動抽出
- 記事の骨格を構築（導入 + 3〜4つの本論 + 結語）
- 5,000〜8,000字の長文を執筆
- 各パートに実体験の裏付けがあり、空虚な意見表明にならない

**3つのSkillsを繋げると、自動化されたコンテンツパイプラインが出来上がる。**

断片的なインプット → `/log` → 構造化日誌（素材抽出付き） → `/tweet` → SNSコンテンツ → `/article` → 深掘り記事

いま読んでいるこの記事が、まさにこのパイプラインの産物だ。過去1ヶ月の日誌に「自律作業」「context蓄積」「起動力」などのテーマが蓄積され、それを `/article` で一本の記事にまとめた。

ここに、もう少し大きな示唆がある。

**Go-to-Marketの形が変わりつつある。**

従来のマーケティングには、チームが必要だった。マーケ部がコンテンツを書き、デザイナーが素材を作り、運用担当がチャネルに流す。1本の記事が選題から公開まで1週間かかることも珍しくなかった。

今は？ビルダー一人ひとりが、自分自身の実践プロセスをコンテンツに変え、継続的に発信することで信頼と影響力を構築できる。マーケチームも外部ライターも不要。日々の仕事そのものが最高のコンテンツ素材になる。

この記事自体がその証明だ。

### 第四幕：自律作業——AIにAIをレビューさせる

第二幕のボトルネックに戻ろう。AIのアウトプットが増えすぎて、人間のレビューが追いつかない。どうするか？

答えはシンプルだった。**AIにAIをレビューさせる。**

2月末、Claude Code + Codex CLIによる「自律作業」ワークフローを構築した。考え方は単純だ。

- **Claude Code** が思考と実行を担当。要件理解、設計、コーディング、ドキュメント作成
- **Codex CLI** が品質管理を担当。コードレビュー、テスト実行、問題の検出
- レビューに通らなければ？自動で差し戻し。修正して再提出。合格するまでループ

日本の製造業が長年培ってきた品質管理のプロセス——「作る人」と「検査する人」を分けるという原則——を、AI同士の協働に適用したようなものだ。

具体的には2つのSkillsに分かれている。

**`/plan` —— 方針策定＋自動レビュー**

Claude Codeと要件について対話する。何を作るか？誰のためか？どんな課題を解決するか？技術スタック、制約条件は？認識合わせが済んだら、Claude Codeが技術方針書を作成する。

そして——ここがポイント——方針書が自動的にCodexのレビューに提出される。

Codexは「シニアテクニカルアーキテクト」として、5つの観点で方針を審査する。アーキテクチャの妥当性、要件カバレッジ、リスク評価、実現可能性、テスト戦略。

審査結果は2種類。`APPROVED`（合格）か `NEEDS_REVISION`（要修正）。

要修正の場合、Codexが指摘事項を項目別にリストアップ。Claude Codeがそれを読み、項目ごとに方針書を修正し、再提出する。

最大5ラウンドの自動ループ。

私が関与するのは最初の要件対話だけ。その後の「方針作成→審査→修正→再審査」のサイクルは完全自動だ。

**`/develop` —— 並行開発＋自動テスト**

方針が通ったら、`/develop` を入力する。

Claude Codeが方針書のモジュール分割に基づきAgent Teamを編成——たとえばフロントエンドagent、バックエンドagent、テストagent——で並行開発。

開発完了後、自動的にCodex Review + Testのループに入る。

1. Codexがコードレビュー——正確性、セキュリティ、パフォーマンス、コード品質、設計との整合性を検査
2. Codexがテスト実行——型チェック、リント、ユニットテスト、E2Eテスト
3. レビューもテストも合格？終了。問題あり？項目をリストアップし、Claude Codeが自動修正、再提出
4. 最大8ラウンド

開発プロセス全体を通じて、すべての意思決定、レビュー記録、バグ記録が `.claude/memory/MEMORY.md` に自動書き込みされる。これ自体がcontextの蓄積となり、フライホイールを形成する。

**実戦での効果は？**

2月28日、このワークフローであるプロダクトのフレームワークをゼロから構築した。全工程自動、バグゼロ。冒頭で描いた風景——私は映画を観ていて、AIたちが勝手に回していた——がこれだ。

**しかもコーディングだけの話ではない。**

同じ考え方が調査やドキュメント業務にも適用できる。一方のagentに調査とドキュメント作成をさせ、もう一方のagentに情報の正確性、論理の完全性、見落としている視点がないかを検査させる。

先ほどの「戦略提携方針」の場面に戻れば——もしClaude Codeに方針書を書かせるだけでなく、Codexに方針書を審査させていたら、「深夜に2時間かけて自分で検収する」という工程も大幅に圧縮できていたはずだ。

**検収の仕事が、人からAIへ移った。**

これが「自律作業」が解決する本質的な問題だ。単体のAIのアウトプット品質には上限がある。2つのAIの対抗がその上限を突破する。人は「実行者」と「検査者」の二重役割から解放され、最初に方向を示し、最後に結果を確認するだけで済む。

### 第五幕：効率が変わると、「野心」も変わる

この1ヶ月を振り返って、いくつか正直な実感がある。

**効率の変化は「何パーセント速くなった」ではなく、「以前は考えもしなかったことに手を出せるようになった」だ。**

「一人でブランド戦略、プロダクト開発、事業提携、コンテンツ制作を同時に進める」——以前なら非現実的だった。今はそれが日常になっている。自分が急に有能になったわけではない。実行フェーズがAIに移ったからだ。

**「野心」が膨張している。**

実行コストがほぼゼロになると、やりたいことが急に増える。以前は「時間がない」で止まっていたアイデアが、今はどれも極めて低コストでまず一版走らせて検証できる。あるマーケター向けのAgentを作ってみたい？AIにフレームワークを組ませてみよう。ある分野のエキスパートの方法論を研究して自社に適用したい？20分でAIに整理させよう。日々の仕事の経験を記事にしたい？`/log` + `/article` で完了。

以前は「実行する余力がない」で死んでいたアイデアが、今は生存率が劇的に上がった。

**「スーパー個人」の再定義。**

「スーパー個人」とは一人ですべてをこなす人ではない。それは超人であって、スーパー個人ではない。

スーパー個人とは、**一人でより多くのことを起動できる人**のことだ。AIが実行チーム、自分は「起動者」——機会を見つけ、判断を下し、方向を示す。そしてAIがそれぞれの方向で結果を出してくれる。

最強のエンジニアである必要も、最高のライターである必要も、最も専門的なアナリストである必要もない。必要なのは、**何をやる価値があるかを見極め、起動する勇気を持ち、実行をAIに任せること。**

**最後に、少し大きなことを言う。**

これらのツールは、仕事を楽にするためのものではない。正直に言えば、私は以前より忙しくなった。できることが増えたからだ。

本当の価値は、**以前はやりたくてもできなかったことに手が届くようになったこと**にある。

「時間がない」「コードが書けない」「一人ではこんなに多くのことはできない」——こうした壁に阻まれていた可能性が、今は阻まれなくなった。

AI時代が「人」にもたらす意味はこれだと思う。仕事を代替するのではない。人生の帯域幅を拡張する。より多くを体験し、より多くを試し、より多くを創れる。

効率化ツールではない。人生の拡張パックだ。

だから、コードが書ける必要はない。Claude Code CLIは対話形式で動く。技術の専門家である必要もない。必要なのは週末の数時間と、「ちょっと試してみよう」という好奇心だけだ。

以下に、すべてのSkillsソースファイルを公開する。コピーすればすぐ使える。

### 付録：Skills全ソースコード＆導入ガイド

#### 事前準備

#### Skill 1：`/log` —— 断片メモ → 構造化日誌

**ファイルパス**：`.claude/commands/log.md`

**使い方**：`/log 今日はXXXをやった、XXXの問題があった、XXXというアイデアが浮かんだ`

```markdown
# 業務日誌の記録

ユーザーが提供する業務内容やメモを、構造化された業務日誌に展開する。

## 実行ステップ

1. **現在の日付と曜日を取得**：`date` コマンドで正確な日付と曜日を取得（例：`date '+%Y-%m-%d %A'`）。**曜日を自分で推測してはならない**。必ずシステムコマンドの結果を使用
2. **ユーザー入力の解析**：業務内容、成果、課題、気づきなどを識別
3. **日誌内容の展開**：
   - 短い記述を完全な段落に展開
   - 可能な文脈や詳細を補足
   - 事実に基づき、ユーザーが言及していない情報を捏造しない
4. **テンプレートに沿って整理**：`templates/daily-log.md` のテンプレート構造を参照
5. **日誌を保存**：
   - パス：`logs/YYYY-MM/YYYY-MM-DD.md`
   - ディレクトリが存在しなければ先に作成
   - 当日の日誌が既に存在する場合、追記か上書きかをユーザーに確認

## 日誌の構成

- 基本情報（日付、曜日）
- 本日の成果（主な完了事項、プロジェクト進捗）
- 学びと気づき（新しい知識/スキル、経験から得た教訓）
- 直面した課題（課題の説明、解決策）
- 明日の予定（優先事項、フォローアップ事項）
- 考察と所感（深い思考や気づき）
- コンテンツ化できる素材（SNS投稿に使える視点、記事に深掘りできるテーマ）

## 執筆の原則

- 中国語で執筆（※日本語で使用する場合は「日本語で執筆」に変更）
- ユーザー本来の表現スタイルを維持
- 適度に展開するが、過度な修飾はしない
- 価値のある洞察ポイントを強調

## ユーザー入力

$ARGUMENTS
```

> **日本語環境で使う場合のヒント**：「中国語で執筆」の部分を「日本語で執筆」に変更するだけで、日本語の業務日誌として機能します。

#### Skill 2：`/tweet` —— 日誌 → SNSコンテンツ（2バージョン）

**ファイルパス**：`.claude/commands/tweet.md`

**使い方**：`/tweet AI時代の起動力` や `/tweet 自律作業ワークフローの実践`

```markdown
# SNSコンテンツの生成

指定されたテーマに基づき、300字の長文版と140字の短文版の2バージョンを生成する。

## 実行ステップ

1. **テーマの理解**：
   - ユーザーが指定したテーマや視点を解析
   - 必要に応じて関連する業務日誌から素材を抽出

2. **素材の検索**：
   - `logs/` 内でテーマに関連する内容を検索
   - 実体験、データ、ケースを抽出

3. **投稿の執筆**：
   - 核心的な視点を際立たせる
   - 実体験で裏付ける
   - 結びは示唆的、または思考を促す内容に

4. **推敲**：
   - 各テーマ/角度ごとに2バージョンを生成：
     - **300字版**（280〜320字）：完全な叙事。Note、LinkedIn等の長文プラットフォーム向け
     - **140字版**（120〜140字）：凝縮されたエッセンス。X/Twitter向け（1投稿の上限140字）
   - 2バージョンはそれぞれ独立した文章とし、短縮版は長文版の単なる削りではない
   - 2〜3つの角度/テーマを提示し、各角度に300字版と140字版を含む

5. **ファイル保存**：
   - パス：`tweets/YYYY-MM-DD-topic.md`
   - 正式版と代替案を含む

## 投稿の構成

- **冒頭（フック）**：注意を引く最初の一文
- **本体**：核心的視点 ＋ 経験/ケースによる裏付け
- **結び**：示唆的な思考、またはアクションの呼びかけ

## 執筆の原則

- 1つの投稿で焦点を当てるのは1つの核心的視点のみ
- 個人の実体験で信頼性を高める
- 説教は避け、ストーリーと具体例で語る
- 自然な言葉遣い。過度にフォーマル・マーケティング的にしない
- 適度な余白を残し、読者に考える余地を与える

## テーマ

$ARGUMENTS
```

- **深みがある**：情報の転載ではなく、独自の洞察
- **ケースがある**：実体験で視点を裏付ける
- **方法がある**：読者に実行可能なアドバイスを提供
- **温度がある**：個人の視点、誠実な表現
- **リズムがある**：長短の文を織り交ぜ、段落は適度な長さ

$ARGUMENTS

## 1. Agent Teams並行開発

`.claude/plans/current-plan.md` のモジュール分割に基づきAgent Teamを編成。

すべてのteammateのルール：
- 開始前に .claude/memory/MEMORY.md を必ず読む

- 独立した機能が完成するごとにgit commit

**すべてのteammateの開発完了を待つ。**

## 2. Codex Review + Testループ

開発完了後、自動ループ：Codex審査＋テスト → 問題があれば修正 → 再提出。全パスするまで。最大8ラウンド。

` ` `bash
MAX_ROUNDS=8
ROUND=0
ALL_PASS=false

while [ $ROUND -lt $MAX_ROUNDS ] && [ "$ALL_PASS" = false ]; do
  ROUND=$((ROUND + 1))
  echo ""
  echo "Codex Review & Test — 第 ${ROUND}/${MAX_ROUNDS} ラウンド"

  # Part A: Code Review
  echo "Code Review..."

  DIFF_STAT=$(git diff main --stat 2>/dev/null || git log --oneline -10 2>/dev/null || echo "No diff")
  CHANGED_FILES=$(git diff main --name-only 2>/dev/null || git diff HEAD~10 --name-only 2>/dev/null || echo "")

  CODE_CONTENT=""
  for f in $CHANGED_FILES; do
    if [ -f "$f" ]; then
      CODE_CONTENT+="
=== FILE: $f ===
$(head -200 "$f")
"
    fi
  done
  CODE_CONTENT=$(echo "$CODE_CONTENT" | head -c 8000)

  REVIEW_RESULT=$(codex exec "You are a senior code reviewer. Review these code changes.

## Changed Files Summary
${DIFF_STAT}

## Code Content
${CODE_CONTENT}

## Review Criteria
1. Correctness: logic errors, edge cases, null/undefined handling
2. Security: input validation, injection risks, auth checks
3. Performance: N+1 queries, re-renders, memory leaks
4. Code Quality: naming, DRY, error handling, type safety
5. Architecture: does implementation match intended design?

## Response Format
Start with EXACTLY one of:
VERDICT: APPROVED
VERDICT: NEEDS_FIX

If NEEDS_FIX, list each issue as:
[CRITICAL/MAJOR/MINOR] file:line — description — suggested fix" \
    --full-auto --sandbox read-only --skip-git-repo-check 2>&1) || true

  echo "$REVIEW_RESULT" > ".claude/reviews/code-review-r${ROUND}.md"

  REVIEW_OK=false
  if echo "$REVIEW_RESULT" | grep -qi "VERDICT:.*APPROVED"; then
    REVIEW_OK=true
    echo "  Code Review PASSED"
  else
    echo "  Code Review: 問題あり"
  fi

  # Part B: テスト実行
  echo "Running Tests..."

  TEST_RESULT=$(codex exec "You are a QA engineer. Run ALL available tests in this project.

Instructions:
1. Check what test frameworks are configured (package.json, pyproject.toml)
2. Run all available checks:
   - npm run typecheck (if exists)
   - npm run lint (if exists)
   - npm test / npx jest / npx vitest run
   - npx playwright test (if playwright.config exists)
   - python -m pytest (if Python tests exist)
3. If no tests exist yet, report that clearly

## Response Format
Start with EXACTLY one of:
